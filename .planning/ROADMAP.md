## Phase Details

### Phase 7: File-Layout Drift Detector
**Goal:** A plugin-local script catches dangling `@~/.claude/...` references in skills/agents/references before they ship. CI enforces.
**Depends on:** Nothing new (runs against the current plugin layout).
**Requirements:** DRIFT-01, DRIFT-02 (file-layout portion)
**Success Criteria (what must be TRUE):**
  1. A detector script (`bin/maintenance/check-file-layout.cjs` or similar) scans all SKILL.md, agent.md, and reference files for `@~/.claude/...` and `@$HOME/.claude/...` path references.
  2. For each reference, the detector verifies the target file exists in the plugin's own layout (not just on the user's disk — because `~/.claude/get-shit-done/` references always exist for the plugin author but never for end users).
  3. When a dangling reference is found, the detector exits non-zero with a clear report (file, line, path that doesn't resolve).
  4. CI invokes the detector; a failing detector blocks merge.
  5. The resume-project.md gap from Phase 5 would have been caught by this detector retroactively.

### Phase 8: HANDOFF Schema Baseline and Detector
**Goal:** A committed `schema/handoff-v1.json` defines the HANDOFF contract; `checkpoint.cjs` output validates against it in CI; a post-sync check compares upstream's `pause-work` output structure against our schema to surface drift.
**Depends on:** Phase 7 (consistent drift-detector pattern established), and research R-1 (upstream `pause-work` output reconnaissance) pulled in at plan time.
**Requirements:** SCHEMA-01, SCHEMA-02, SCHEMA-03, DRIFT-02 (schema portion)
**Success Criteria (what must be TRUE):**
  1. `schema/handoff-v1.json` exists, version-suffixed, and describes the 19-field contract from `bin/lib/checkpoint.cjs` with field types + required/optional status.
  2. CI validates `checkpoint.cjs` output against the schema on every commit.
  3. A post-sync check (invokable as `node bin/maintenance/check-handoff-schema.cjs --compare-upstream`) reads upstream's `/gsd:pause-work` skill, extracts the structure it produces, and diffs against the committed schema.
  4. Detected drift fails loud with a structured diff report (added fields, removed fields, type changes).
  5. Research R-1 has been answered — the scope of "compare against upstream" is clearly bounded (either fixed point or tracked-moving-target).

### Phase 9: Unified Check + Docs + Post-Sync Integration
**Goal:** One entry-point for all drift checks; docs that explain the drift story; post-sync workflow updated to require the check.
**Depends on:** Phases 7 + 8
**Requirements:** DRIFT-03, DOCS-01, DOCS-02, MAINT-01, DRIFT-02 (integration portion)
**Success Criteria (what must be TRUE):**
  1. `bin/maintenance/check-drift.cjs` runs file-layout + schema + namespace detectors in sequence and reports a consolidated summary.
  2. README.md contains a paragraph describing session continuity (the feature) and drift resilience (how the plugin protects itself + the user).
  3. `CHANGELOG.md` exists with a v2.38.2 entry for v1.1 shipment and working-forward entries for v1.2 phases.
  4. PROJECT.md's "After each upstream GSD sync" checklist includes running `node bin/maintenance/check-drift.cjs` before declaring the sync complete.
  5. Running `check-drift.cjs` against the current committed tree returns 0 (clean baseline).
