## Deviations from Plan

### 1. [Fallback path chosen for Task 4 — pre-authorized by plan risk section]

- **Found during:** Task 4, first attempt at preferred path (insert placeholder markers → run `generate-claude-md` to populate).
- **Issue:** After inserting placeholder markers and running `node bin/gsd-tools.cjs generate-claude-md`, the command succeeded but rewrote the project/stack/conventions/architecture sections in CLAUDE.md — pulling content from PROJECT.md / STACK.md / CONVENTIONS.md / ARCHITECTURE.md files that have drifted from the minimal "Loaded on demand by GSD commands." stubs the repo currently ships. Diff was 299 insertions / 11 deletions across 4 unrelated sections. Way outside the scope of plan 05-01 which only owns session-continuity.
- **Plan coverage of this case:** The plan's `<risks>` section pre-authorized the fallback: *"If the existing CLAUDE.md has drifted from the managed template, the regeneration may rewrite more than intended. Mitigation: diff-review the CLAUDE.md change before committing; if it looks wrong, fall back to a targeted insertion."* The fallback path is also explicitly documented inside Task 4's `<action>` block.
- **Fix:** Reverted the regeneration (`git checkout -- CLAUDE.md`), then hand-inserted the exact marker-wrapped Session Continuity section body between the `architecture-end` and `workflow-start` markers. Content matches `CLAUDE_MD_SESSION_CONTINUITY` byte-for-byte (532-byte section body verified by a length + exact-equality check extracting the constant at runtime from the source module and comparing to the extracted section body).
- **Idempotency criterion:** Task 4's second automated verify (`generate-claude-md` run twice should produce identical hashes) is NOT satisfied — running it would resurrect the unwanted regeneration of the four unrelated sections. Task 2's registration verification separately confirms the generator round-trip works correctly end-to-end: the constant is wired into MANAGED_SECTIONS/generators/sectionHeadings, and the generator returns the exact same byte-content that is now hand-placed in CLAUDE.md. A cold-start user generating a new project CLAUDE.md from scratch will get the section in the correct position automatically.
- **Files modified:** CLAUDE.md (single section insertion — no other content touched).
- **Commit:** `cde38f4`
- **Follow-up:** The drift between the minimal CLAUDE.md stubs and the actual PROJECT.md/STACK.md/CONVENTIONS.md/ARCHITECTURE.md source files is a separate repo hygiene issue — either the stubs should be regenerated from the sources deliberately (as a quick task) or the sources should be renamed so `generate-claude-md` ignores them. Not blocking for BKUP-01/BKUP-02.

### 2. [Process deviation — accidental mixed commit, then split]

- **Found during:** Initial Task 4 commit (ultimately discarded as `a6dbe95`).
- **Issue:** Plan 02's executor is running in parallel in the same working copy. At the moment my `git commit` ran for Task 4, plan 02 had already staged its SUMMARY.md and .planning/ state files for its own final commit. My `git add CLAUDE.md && git commit` swept all staged files into one commit mislabeled `chore(05-01)`.
- **Fix:** `git reset --soft HEAD~1` to unstage, `git restore --staged` to remove plan 02's files from the index (files remain on disk for plan 02 to commit), then re-committed with only CLAUDE.md staged → clean commit `cde38f4`.
- **Not a Rule 1-4 deviation from the plan itself** — this is a working-copy coordination artifact from parallel execution. No code impact.
- **Observation for future parallel plan scheduling:** When multiple executors share a working copy, `git add <paths> && git commit <paths>` (commit with explicit paths) is safer than `git add <paths> && git commit` (commit of whatever is staged). Both plans now committed independently.
